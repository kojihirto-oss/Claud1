# glossary（用語の唯一定義）

## 目的
このフォルダは **用語の揺れを防止** し、同じ概念を複数の表記で呼ばないための **唯一の用語集** を管理する。

## 運用ルール（MUST）

### 1. 用語の追加
新しい概念・略語・専門用語が登場したら、**即座に glossary/ に追加** する。

**追加すべき用語の例**:
- プロジェクト固有の略語（VCG, VIBE, SSOT など）
- 多義的な用語（「リリース」「ビルド」「デプロイ」など）
- 類義語が多い概念（「検証」「確認」「テスト」など）
- 技術用語の日英対応（「認証/Authentication」など）

### 2. 用語の統一
docs/ で用語を使う際は、**必ず glossary/ の表記に統一** する。

**禁止事項**:
- glossary/ にない表記を勝手に使う
- 同じ概念を複数の表記で呼ぶ（例: 「チェック」「検証」「確認」を混在）
- 口語・略称を docs/ で使う（例: 「OK」→「合格」、「NG」→「不合格」）

### 3. 揺れの修正
用語の揺れを発見したら：
1. glossary/ を確認し、正式な表記を特定
2. docs/ の該当箇所を修正
3. checks/ で用語揺れ検出を追加（推奨）

## 命名規則

### ファイル名
```
GLOSSARY.md
```
- 最初は単一ファイルで管理
- 用語が100を超えたら、トピック別に分割可（例: `GLOSSARY_SECURITY.md`, `GLOSSARY_INFRA.md`）

### 内容形式
```markdown
## 用語名

**定義**: （簡潔な説明）

**参照Part**: Part02, Part05

**類義語**: （他の表現、使ってはいけない表記）

**例**: （使い方の例）
```

**記載例**:
```markdown
## SSOT (Single Source of Truth)

**定義**: 唯一の正本。複数のコピーを持たず、1箇所だけを更新・参照する運用。

**参照Part**: Part00, Part14

**類義語**: マスターデータ、正本（「マスタ」「原本」は使わない）

**例**: 「docs/ が SSOT であり、sources/ は材料であって SSOT ではない」
```

## チェックリスト（用語追加時）
- [ ] 用語名が明確（略語の場合、正式名称も記載）
- [ ] 定義が簡潔（1-2文）
- [ ] 参照Part が明記されている
- [ ] 類義語・禁止表記が明記されている（該当する場合）
- [ ] docs/ で該当用語を使っている箇所を確認し、表記を統一

## 検証方法
checks/ に以下を追加することを推奨：
```bash
# 用語揺れ検出（例）
grep -r "マスタ" docs/  # 「SSOT」に統一すべき
grep -r "チェック" docs/  # 「検証」に統一すべき
```
