#### 新規セクションまたは既存Part09への追記

### 5.X MCP Server/Client間のセキュリティ【追加】

#### 5.X.1 Confused Deputy攻撃の防止【MUST】

**脆弱条件**:
以下の条件がすべて揃うと、Confused Deputy攻撃が成立する：
- MCP proxyサーバーがサードパーティ認可サーバーに対して**静的クライアントID**を使用
- MCP proxyサーバーがMCPクライアントの**動的登録**を許可（各クライアントが独自のclient_idを取得）
- サードパーティ認可サーバーが初回認可後に**同意Cookie**を設定
- MCP proxyサーバーがサードパーティ認可へ転送する前に**クライアントごとの同意**を実装していない

**攻撃フロー**:
1. ユーザーが正規のMCP proxyサーバー経由でサードパーティAPIに認証
2. サードパーティ認可サーバーが静的クライアントIDに対する同意Cookieをユーザーエージェントに設定
3. 攻撃者が悪意のあるリダイレクトURIを含む認可リクエストをユーザーに送信（新規動的登録クライアントIDを使用）
4. ユーザーがリンクをクリック時、ブラウザに残る同意Cookieにより同意画面がスキップされる
5. MCP認可コードが攻撃者のサーバーへリダイレクトされる
6. 攻撃者がユーザーの明示的承認なしに認可コードをアクセストークンに交換

**緩和策**（MCP proxyサーバーはMUSTで実装）:

##### Per-Client Consent Storage【MUST】
- ユーザーごとに承認済み`client_id`のレジストリを維持
- サードパーティ認可フローを開始する**前**にこのレジストリをチェック
- 同意決定をサーバー側（データベースまたはサーバー固有Cookie）にセキュアに保存

##### Consent UI Requirements【MUST】
MCPレベルの同意ページは以下を満たす：
- リクエスト元MCPクライアントを名前で明確に識別
- リクエストされるサードパーティAPIスコープを表示
- トークンが送信される登録済み`redirect_uri`を表示
- CSRF保護（state parameter、CSRFトークン等）を実装
- `frame-ancestors` CSPディレクティブまたは`X-Frame-Options: DENY`でiframingを防止

##### Consent Cookie Security【MUST】
同意決定を追跡するためにCookieを使用する場合：
- Cookie名に`__Host-`プレフィックスを使用
- `Secure`、`HttpOnly`、`SameSite=Lax`属性を設定
- 暗号学的に署名するか、サーバー側セッションを使用
- 特定の`client_id`にバインド（単なる「ユーザーが同意した」ではなく）

##### Redirect URI Validation【MUST】
- 認可リクエスト内の`redirect_uri`が登録済みURIと**完全一致**することを検証
- `redirect_uri`が再登録なしに変更された場合はリクエストを拒否
- 完全文字列一致を使用（パターンマッチングやワイルドカード禁止）

##### OAuth State Parameter Validation【MUST】
OAuth `state`パラメータは認可コード傍受とCSRF攻撃を防ぐために重要：
- 各認可リクエストに対して暗号学的にセキュアなランダム`state`値を生成
- `state`値をサーバー側（セキュアなセッションストアまたは暗号化Cookie）に保存するのは同意が明示的に承認された**後のみ**
- `state`追跡Cookie/セッションをサードパーティIDプロバイダーへリダイレクトする**直前**に設定（同意承認前ではない）
- コールバックエンドポイントで、`state`クエリパラメータがコールバックリクエストのCookieまたはセッションに保存された値と完全一致することを検証
- `state`パラメータが欠落または不一致の場合はコールバックリクエストを拒否
- `state`値は単一使用（検証後削除）かつ短い有効期限（例: 10分）を持つ

**重要**: `state`値はMCPサーバーの認可エンドポイントでユーザーが同意画面を承認した**後**にのみ設定すること。同意承認前にこのCookieを設定すると、攻撃者が悪意のある認可リクエストを細工することで同意画面を迂回できるため、同意画面が無効化される。

**根拠**: KB_MCP_SECURITY_20260115_112146.txt (6694-6833行)
**参照**: https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices
```

## 2. MCP Security: Token Passthrough禁止と監査証跡

### GAP
既存Part09/Part10には、**トークンパススルー（Token Passthrough）の禁止**と、それに伴う**監査証跡の欠落リスク**が明記されていない。

### ADD

#### 既存Part09への追記

```markdown
### 5.X.2 Token Passthrough禁止【MUST NOT】

**定義**:
Token Passthroughは、MCPサーバーがMCPクライアントからトークンを受け取り、そのトークンがMCPサーバー宛に適切に発行されたかを検証せず、ダウンストリームAPIにパススルーするアンチパターン。

**リスク**:
- **セキュリティコントロールの迂回**: MCPサーバーまたはダウンストリームAPIが実装するレート制限、リクエスト検証、トラフィック監視などのセキュリティコントロールをバイパス
- **説明責任と監査証跡の問題**:
  - MCPサーバーがMCPクライアントを識別/区別できない（上流発行のアクセストークンがMCPサーバーに対して不透明な場合）
  - ダウンストリームResource Serverのログに、実際にトークンを転送しているMCPサーバーではなく、異なるソースや異なるIDから来たように見えるリクエストが記録される
  - 両方の要因により、インシデント調査、コントロール、監査が困難になる
  - MCPサーバーがトークンのクレーム（ロール、権限、audienceなど）やメタデータを検証せずにパスする場合、盗まれたトークンを所持する悪意のあるアクターがサーバーをデータ流出のプロキシとして使用できる
- **信頼境界の問題**: ダウンストリームResource Serverが特定のエンティティに信頼を付与している前提が崩れ、予期しない問題が発生
- **将来互換性リスク**: MCPサーバーが将来セキュリティコントロールを追加する際、適切なトークンaudience分離から始めることでセキュリティモデルの進化が容易になる

**ルール**:
- **MUST NOT**: MCPサーバーは、MCPサーバー宛に明示的に発行されたトークン以外を受け入れてはならない
- **MUST**: MCPサーバーはトークンのaudience、issuer、有効期限、スコープを検証する
- **MUST**: MCPクライアントからの認証は、MCPサーバーが発行または検証可能なトークンに限定する

**根拠**: KB_MCP_SECURITY_20260115_112146.txt (6834-6856行)
**参照**: MCP Authorization Specification (Token Passthrough禁止)
```

```markdown
## 9.X 監査ログのストレージ容量管理【追加】

### 9.X.1 AU-4: Audit Storage Capacity（NIST SP 800-53準拠）【MUST】

**要件**:
組織定義の監査記録保持要件に対応した監査記録ストレージ容量を割り当てる。

**実装**:
- **MUST**: evidence/verify_reports および evidence/humangate_approvals の想定データ量を事前に見積もる
  - 1回のVerify実行 = 4ファイル（Fast）または追加ファイル（Full）
  - 1セットあたりの平均サイズを計測（例: Fast = 50KB、Full = 500KB）
  - 年間Verify実行回数の見積もり（例: 1日10回 × 365日 = 3,650回）
  - 年間必要容量 = 3,650回 × 50KB = 182.5MB
- **MUST**: ストレージ容量を監視し、以下の閾値でアラートを発行
  - 警告: 容量の70%使用時
  - クリティカル: 容量の90%使用時
- **MUST**: 容量超過を防ぐため、以下のアーカイブ戦略を実施
  - 直近3セットを evidence/verify_reports に保持（Part10で定義済み）
  - 4セット目以降は evidence/archive/YYYY/MM/ へ移動（削除禁止）
  - アーカイブ後、圧縮（zip/tar.gz）してストレージ効率を向上
- **SHOULD**: 年次レビューで保持ポリシーを見直し、必要に応じて容量を拡張

**監視スクリプト例**（checks/配下に追加）:
```powershell
# checks/monitor_audit_storage.ps1
$auditPath = "evidence/verify_reports"
$archivePath = "evidence/archive"
$maxSizeGB = 10  # 最大10GB

$currentSizeGB = (Get-ChildItem -Path $auditPath -Recurse | Measure-Object -Property Length -Sum).Sum / 1GB
$usagePercent = ($currentSizeGB / $maxSizeGB) * 100

if ($usagePercent -ge 90) {
    Write-Error "CRITICAL: Audit storage at $usagePercent% capacity"
} elseif ($usagePercent -ge 70) {
    Write-Warning "WARNING: Audit storage at $usagePercent% capacity"
} else {
    Write-Output "OK: Audit storage at $usagePercent% capacity"
}
```

**根拠**: KB (NIST SP 800-53 AU-4, 5316-5322行)
**参照**: https://csrc.nist.gov/pubs/sp/800/53/r5/upd1/final
```

## 4. 監査ログ: 証跡の再現性と相関ID

### GAP
既存Part09/Part10には、**監査ログの相関ID**（どのユーザーがどの操作を実行したか追跡）および**証跡の再現性**（ログからインシデントを再現可能）が未定義。

#### 既存Part09への追記

```markdown
### 9.X.3 監査ログの相関ID【MUST】

**目的**:
複数システム・コンポーネント間の操作を追跡可能にし、インシデント調査時に「誰が、いつ、何を、どのように実行したか」を特定する。

**実装**:
- **MUST**: 以下のメタデータをすべての監査ログに含める
  - **Correlation ID**: UUID v4形式の一意識別子（操作全体を追跡）
  - **User ID**: 実行者（AI Agent: `claude-code@ssot`, Human: `user@example.com`）
  - **Timestamp**: ISO 8601形式（例: `2026-01-16T08:52:39+09:00`）
  - **Operation Type**: 操作種別（`verify_fast`, `humangate_approval`, `git_commit`等）
  - **Target**: 操作対象（ファイルパス、Part番号、ADR番号等）
  - **Result**: 成功/失敗（`PASS`, `FAIL`, `APPROVED`, `REJECTED`）
  - **Context**: 追加コンテキスト（変更差分、承認理由、エラーメッセージ等）

**フォーマット例**（JSON Lines形式）:
```json
{"correlation_id":"a3b8c9d0-1234-5678-9abc-def012345678","user_id":"claude-code@ssot","timestamp":"2026-01-16T08:52:39+09:00","operation":"verify_fast","target":"docs/Part10.md","result":"PASS","context":"4 checks passed"}
{"correlation_id":"a3b8c9d0-1234-5678-9abc-def012345678","user_id":"claude-code@ssot","timestamp":"2026-01-16T08:53:12+09:00","operation":"git_commit","target":"docs/Part10.md","result":"SUCCESS","context":"commit_hash:abc123"}
{"correlation_id":"b4c9d0e1-2345-6789-abcd-ef0123456789","user_id":"user@example.com","timestamp":"2026-01-16T09:00:00+09:00","operation":"humangate_approval","target":"decisions/0005-add-mcp-security.md","result":"APPROVED","context":"SLA:within_24h"}
```

**保存先**:
- `evidence/audit_log/YYYYMMDD.jsonl`（日次ローテーション）
- 30日分を保持、それ以降は `evidence/archive/audit_log/YYYY/MM/` へ移動

**検索例**:
```powershell
# 特定のCorrelation IDで操作を追跡
cat evidence/audit_log/20260116.jsonl | jq 'select(.correlation_id=="a3b8c9d0-1234-5678-9abc-def012345678")'

# 特定ユーザーの操作を抽出
cat evidence/audit_log/20260116.jsonl | jq 'select(.user_id=="claude-code@ssot")'

# 失敗した操作を抽出
cat evidence/audit_log/20260116.jsonl | jq 'select(.result=="FAIL" or .result=="REJECTED")'
```

**根拠**: KB (Token Passthrough監査証跡問題 6842-6845行、Zero-trust audit logging 23167-23173行)
```

### RISK

### 6.3.X HumanGate承認フローの詳細【拡張】

#### 6.3.X.1 承認ステージ【MUST】

**Pre-deployment承認**（デプロイ前承認）:
- **タイミング**: 変更を docs/ に反映する前
- **目的**: 破壊的変更、ADR追加、Part番号変更等の事前審査
- **承認者**: decisions/0004-humangate-approvers.md で定義された主要承認者
- **タイムアウト**: 通常24時間、重要案件48時間、緊急案件2時間
- **承認ポリシー**:
  - 単一承認者: 1名の承認で進行可（軽微な変更）
  - 複数承認者: 2名以上の承認が必要（破壊的変更）
  - 自己承認禁止: 変更実施者は承認者になれない（デフォルト）

**Post-deployment承認**（デプロイ後承認）:
- **タイミング**: 変更を docs/ に反映した後、git push 前
- **目的**: 変更結果の確認、Verify PASS の妥当性審査
- **承認者**: decisions/0004-humangate-approvers.md で定義された代理承認者
- **タイムアウト**: 12時間
- **承認ポリシー**:
  - 自動承認: Pre-deployment承認者と同一ユーザーの場合、自動承認可（重複レビュー削減）

#### 6.3.X.2 承認通知【MUST】

- **MUST**: 承認要求時、承認者にメール/チャット通知を送信
  - 件名: `[HumanGate] 承認要求: <操作内容>`
  - 本文: 操作内容、リスク評価、承認期限、承認/却下リンク
- **MUST**: タイムアウト1時間前に再通知
- **MUST**: タイムアウト時、エスカレーション（上位承認者または緊急承認者へ通知）

#### 6.3.X.3 承認ログ【MUST】

- **保存先**: `evidence/humangate_approvals/YYYYMMDD_HHMMSS_<operation>_<status>.md`
- **内容**:
  - 承認要求日時
  - 承認者（または却下者）
  - 承認/却下日時
  - SLA判定（期限内/期限超過）
  - 操作内容（変更ファイル、リスク評価、代替案）
  - 承認理由または却下理由

**例**:

- **承認者**: user@example.com (主要承認者)
- **承認日時**: 2026-01-16 12:30:00
- **SLA判定**: 期限内（24時間以内）
- **操作内容**: decisions/0005-add-mcp-security.md の追加
- **リスク評価**: 新規ADRの追加、既存Partへの影響なし
- **承認理由**: MCP Securityは重要テーマであり、追記が必要と判断

### 5.7.1 Required Status Checks【MUST】

**目的**:
mainブランチへのマージ前に、必ず以下のチェックをCI/CDで実行し、全てPASSしない限りマージを禁止する。

**実装**:
- **MUST**: GitHub/GitLab/Azure DevOpsのBranch Protection設定で以下を有効化
  - `Require status checks to pass before merging`（GitHub）
  - `Pipelines must succeed`（GitLab）
  - `Build validation`（Azure DevOps）
- **MUST**: 以下のStatus Checksを必須化
  - `verify-fast`: Fast Verify（4点チェック）
  - `verify-full`: Full Verify（mainマージ時のみ）
  - `forbidden-patterns`: 禁止コマンド検出
  - `sources-integrity`: sources/ 改変検出

**CI/CDワークフロー例**（GitHub Actions）:
```yaml
name: Verify Gate
on:
  pull_request:
    branches: [main, develop]
jobs:
  verify-fast:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run Fast Verify
        run: pwsh ./checks/verify_repo.ps1 -Mode Fast
      - name: Upload evidence
        uses: actions/upload-artifact@v3
        with:
          name: verify-reports
          path: evidence/verify_reports/
```

### 5.7.2 Review Requirements【MUST】

**目的**:
人間によるコードレビューを強制し、AI Agent単独での変更を防止する。

**実装**:
- **MUST**: Branch Protectionで以下を有効化
  - `Require pull request reviews before merging`（GitHub）
  - 必要承認数: 1名以上（破壊的変更は2名以上）
  - `Dismiss stale pull request approvals when new commits are pushed`（新規コミット時に承認をリセット）
- **SHOULD**: Last Reviewer要件を有効化
  - `Require approval of the most recent reviewable push`（GitHub）
  - 最新のpushは、push実施者以外の承認者による承認が必要（攻撃者が最終段階で悪意のあるコードを挿入することを防止）

### 5.7.3 Deployment Gates（CI/CD統合）【SHOULD】

**目的**:
外部サービス（セキュリティスキャン、ログ監視等）からの健全性シグナルを自動収集し、デプロイ可否を判定する。

**実装例**（Azure DevOps Gates）:
- **Pre-deployment Gates**:
  - セキュリティスキャン（SonarQube、Snyk等）がPASS
  - 過去24時間のインシデント数が閾値以下（例: 0件）
  - ストレージ容量が閾値以下（例: 70%未満）
- **Post-deployment Gates**:
  - デプロイ後5分間、エラーログが閾値以下（例: 0件）
  - Verify Fullが全項目PASS

**評価オプション**:
- **Delay before evaluation**: ゲート評価開始前の待機時間（例: 5分）
- **Timeout**: ゲート評価のタイムアウト（例: 30分）
- **Sampling interval**: ゲート再評価の間隔（例: 5分ごと）
- **判定**: タイムアウト前にすべてのゲートが同時にPASSしない限り、デプロイを拒否

### RISK
**追記しない場合の事故シナリオ**:
1. AI Agentが誤って破壊的変更をmainブランチに直接push
2. Verify FAILの状態でマージされ、SSOT全体が破壊される
3. 人間によるレビューがスキップされ、セキュリティ脆弱性が混入
4. Last Reviewer要件がないため、攻撃者が最終コミットで悪意のあるコードを挿入
5. 本番環境デプロイ後、セキュリティインシデントが発生し、全システム停止
